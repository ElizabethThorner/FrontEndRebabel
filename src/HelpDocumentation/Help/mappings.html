Mappings

Due to the difficulty (indeed, probably impossibility) of creating a naming system that covers all use-cases, reBabel expects each format converter to operate in a separate namespace. Thus, the FLExText importer will create features beginning with FlexText: while the CoNLL-U importer creates ones starting with UD:. Likewise, the exporters for those formats check for features with the same prefixes and so a conversion from one file type to another without any parameters will output a mostly empty document.

The solution to this is mappings. Mappings define temporary renamings of types or features during import and export. There are two types of mappings implemented in this interface: type and feature.

Type Mappings
In Type Out Type
sentence phrase

A type mapping establishes an equivalence between two different type names. A type refers to a category particular to a specific file format. In the mapping above, we declare that a sentence in a CoNLL-U file should be treated by the format converter as if it said phrase. So, since the CoNLL-U converters operate on sentence nodes and the FLExText converters operate on phrase nodes, we could use this mapping to convert from CoNLL-U to FLExText.

Feature Mappings
In Feature Out Feature
UD:lemma FlexText:en:lem

Similarly, a feature mapping establishes an equivalence for a particular feature name. A feature refers to the category of a particular instance of data of a particular type. Here we're mapping between the CoNLL-U UD:lemma, of the type "word," and the FLExText FlexText:en:lem, also of type "word." Because these types have the same name, they do not require a type mapping to establish an equivalence.
